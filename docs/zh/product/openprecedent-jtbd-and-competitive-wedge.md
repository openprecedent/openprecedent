## 为什么写这份文档

这份文档用于固化一轮正在进行中的产品讨论：OpenPrecedent 到底应该替用户完成什么待完成任务（Jobs To Be Done，JTBD）。

这里真正要解决的问题，不是 Agent 能不能产生日志、追踪信息（tracing）或记忆检索结果，而是项目演进和 Agent 服务过程中，关键决策信息会持续流失。

这些缺失的信息可能出现在：

- 会议纪要里
- Slack 或其他即时通讯（IM）线程里
- 最终代码变更之外的仓库讨论里
- 客户特异性的例外要求里
- 只能靠专家回忆、却没有被清晰写下来的历史判断里

一旦这些决策信号消失，后续工作就容易滑向“看起来合理、其实错误”的通用解，重复已经否掉的方向，或者忽略仓库特异性和客户特异性约束。

因此，OpenPrecedent 应该围绕这个问题来评估，而不是围绕一个泛化的“Agent 可观测性”分类来评估。

## 当前产品论点

OpenPrecedent 应该成为一层面向 Agent 与操作者的判断继承层。
它的任务不是为了存更多历史而存历史，而是在真实项目约束下需要做出新决策时，保住并取回仍然有用的历史判断。

这件事最重要的地方，恰恰在于关键上下文往往被切碎地分布在运行时历史、人类讨论、客户特异性例外和专家记忆里。
它的战略路径不是一步替代专家，而是在真实的专家与 Agent 协作中不断捕获先例，让可复用判断逐步沉淀，慢慢降低组织对脆弱人工回忆的依赖。

## 通用基础设施与首个切入点

一个反复出现的误区，是把 OpenPrecedent 的长期产品类别，直接等同于它当前最先验证的环境。
这样理解会把产品看得过窄。

OpenPrecedent 不应该被理解成只服务软件团队，或者只服务开发者项目先例管理的基础设施。
更准确地说，它应该是一层面向全部 Agent 的通用决策先例基础设施。
只要 Agent 参与执行，就往往存在决策路径、判断流失，以及后续需要重新利用历史判断的问题。

这个更宽的判断理论上适用于很多领域，例如：

- 软件交付与编码 Agent
- 销售与面向客户的 Agent
- 运营与组织流程 Agent
- 个人助理 Agent
- 工业或运营控制 Agent

这些领域真正共享的，不是表面的工作流，而是更底层的结构：

- Agent 或人与 Agent 的组合，会在局部约束下持续做决策
- 重要判断往往是临时的、分散的，或者在最终产物里表达得很弱
- 如果这些判断能作为先例在下一次相似情境中被取回，就能减少重新发现和重复犯错

但与此同时，首个切入点仍然必须比最终产品类别窄得多。
不同领域的决策类型、捕获表面、证据模型、风险结构和有用性指标差异都很大。
所以产品不能假设，只要某一个初始工作流成立，就自动证明所有领域都同样成立。

因此，更合理的分层表达应该是：

- 产品类别，是面向全部 Agent 的通用决策先例基础设施
- 当前验证切入点，是软件交付与编码 Agent，因为这里最容易具体观察决策路径、捕获证据和验证下游影响
- 未来扩展方向，是在一个高信号环境中证明核心先例模型和检索逻辑后，再进入其他 Agent 密集型领域

这个区分在战略上非常重要。
如果把 OpenPrecedent 只讲成开发者工具，它的 ambition 会被人为压小。
如果只讲成一个抽象的通用 Agent 平台，又会让验证路径失焦，产品论点难以被真正检验。
它需要两者同时成立：既有通用基础设施的 ambition，也有足够具体的首个切入点。

## 什么应该保持通用，什么应该保持领域特异

如果 OpenPrecedent 真的要成为跨领域的 Agent 基础设施，最难的问题其实不是“领域够不够多”，而是系统里哪些部分应该跨领域稳定，哪些部分必须保持可适配。

最容易出错的地方，是把表面工作流强行抽象成通用层。
软件交付、销售、运营、个人助理、工业控制，并不共享同一种工作流。
它们真正可能共享的，是一层更底层的“决策先例内核”。

### 跨领域稳定的对象模型

目前最有希望跨领域稳定的对象包括：

- `case`
  一个有边界的工作单元或情境。
  不同领域里的具体含义会不同，但角色是一致的：它承载了一段需要行动的完整工作。

- `event`
  一个带时间顺序的运行时或交互历史片段。
  它属于原始材料层，而不是判断层。

- `decision`
  一个从事件中抽取出来的记录，表达“在真实约束下，从若干路径中选了一条”。
  这是 OpenPrecedent 和通用事件存储真正拉开差异的核心对象。

- `artifact`
  与决策有关的外部证据、记录或产物，例如代码差异、PR 讨论、聊天消息、会议纪要、合同条款、工单、看板截图或报告。

- `precedent`
  一个被保留下来、值得未来重新引用的历史决策样本。
  它不只是“曾经发生过的一个决策”，而是“被保留到足以让未来判断再次参考的一条历史判断”。

- `applicability`
  一个显式或隐式的边界层，描述这条先例在什么情况下仍相关、什么情况下不相关、哪些约束仍然必须匹配。

- `invalidation` 或 `supersession`
  一类信号，用来表达某条先例后来被替代、被推翻、已过期，或者只在更新信息出现前成立。

这些对象，应该比任何一个领域里的具体分类法更稳定。
如果核心模型过度依赖仓库、PR、议题或编码场景词汇，它就很难平滑地泛化出去。

### 跨领域稳定的决策字段

围绕 `decision` 和 `precedent` 这层，我认为最值得保持跨领域稳定的字段包括：

- `context`
  决策发生时的局部情境。
  它应该保留足够多的周边状态，让后续人或 Agent 能理解当时的压力和范围。

- `candidate_options`
  当时真实存在或被认真考虑过的候选路径。

- `chosen_option`
  最终被选中的路径。

- `default_path`
  如果没有特殊约束，本来最可能被采用的标准路径、主流路径或默认路径。

- `deviation_reason`
  一个简洁说明，解释为什么最终选项偏离了默认路径。

- `constraints`
  真正塑造这次判断的约束条件。
  这类约束可能包括客户特异性要求、审批边界、兼容性限制、安全规则、成本上限、个人偏好、政策约束或时间压力。

- `rejected_reasons`
  其他重要选项为什么没有被采用。
  这很重要，因为未来 Agent 往往需要避免重新发现一遍已经被否掉的路径。

- `decision_actor`
  这次判断主要由谁形成：
  人、人机共同，还是 Agent 主导。

- `authority_boundary`
  这次决策是否触及审批、所有权、责任归属或授权边界。

- `evidence_refs`
  当时依赖了哪些证据、材料或运行时观察。

- `outcome`
  这个决策后来的实际结果如何。
  早期捕获时可以未知，但模型应该允许后续补充。

- `temporality`
  这个决策原本被视为长期方案、临时方案、过渡方案、应急方案，还是明确只在本地生效。

- `applicability`
  这条先例未来在什么情境下仍应被认为相关。

- `superseded_by` 或 `invalidated_by`
  后来的连接信息，用来表达这条先例已经不该按原样继续沿用。

并不是说每次早期捕获都要把这些字段完整填满。
但从语义上看，这些字段比任何领域专有标签，都更接近“可复用先例”的真正骨架。

### 应该保持领域特异的部分

系统里有几类东西，不应该太早被强行压成一套通用 schema：

- 决策分类法
  软件团队会谈架构选型、兼容性权衡、缺陷修复、版本裁剪。
  销售团队会谈折扣、让步边界、审批例外。
  工业场景会谈安全绕行、降级模式、人工控制切换。

- 捕获触发时机
  合适的捕获时刻在不同领域完全不同。
  编码里可能是写入前、规划中或失败恢复后。
  销售里可能是报价前、做承诺前。
  运营里可能是升级前、改路由前或人工干预前。

- 证据来源与连接器
  仓库、聊天系统、工单系统、客户关系管理（CRM）、企业资源计划（ERP）、传感器流、看板、日历和邮件系统差异都非常大。

- 有用性评估指标
  编码流程可能看返工和错误默认实现。
  销售流程可能看利润侵蚀、审批违规和错误承诺。
  工业流程可能看停机时长、安全事故和不稳定绕行。

如果太早把这些东西压平成一个“通用工作流模型”，系统会看起来很通用，实际上却变弱。

### 更合理的产品分层

更有防御力的长期形态，我认为应该是三层：

- `core precedent layer`
  稳定的核心对象模型，负责 case、event、decision、artifact、precedent、适用性和失效关系。

- `domain adaptation layer`
  领域特异的分类法、触发点、连接器和评估逻辑。

- `runtime usage layer`
  在规划、审批、例外处理、失败恢复等具体时刻，把先例真正送到运行中的 Agent 或操作者面前。

如果没有核心层，OpenPrecedent 最后会变成一堆互相独立的垂直流程。
如果没有领域层，它又会变成一个抽象好看但很难落地的空壳。
如果没有运行时层，它就会退化成一个被动归档系统。

### 需要同时避免的两种失真

这里有两种方向相反、但都很危险的抽象失真：

- 过度泛化
  模型看起来很优雅，也能描述很多领域，但它并不能告诉任何一个真实领域应该在哪里捕获、什么最重要、怎样证明有用。

- 过度垂直化
  模型把首个切入点的语言烙得太深，导致后面其他领域一接入就必须扭曲 schema，甚至重做核心对象模型。

设计目标不是证明所有 Agent 领域都一样。
设计目标是找出“先例”最小、最稳定的语义内核，把其他部分尽量留在可适配层。

### 当前的工作性判断

目前最安全的一句工作性判断是：

OpenPrecedent 的首要抽象，不应该是某个领域的工作流，而应该是“在约束下形成的、可被未来再次引用的历史判断”。

这个判断，比直接抽象领域流程，更有机会支持从编码 Agent 向其他 Agent 场景的扩展。

## 产品问题

用户并不是为了拥有一个决策数据库才需要 OpenPrecedent。
用户真正需要的是：可靠地找回并复用过去工作背后的判断结构。

更具体地说，风险不只是 Agent 忘记事实。
更深的风险是 Agent 或后来的操作者忘记了：

- 当时为什么选这条路
- 当时有哪些关键约束
- 哪些方案被认真考虑过并最终否掉
- 哪些审批边界或权限边界曾经起作用
- 哪些客户或项目必须走非标准例外

如果这些信号缺失，后续迭代即使看起来专业，也可能在方向上持续偏离。

## 为什么现有方式会失效

OpenPrecedent 不应该靠“现有系统什么都记录不了”来证明自己存在。
更准确的说法是：现有系统能记录很多相邻信息，但它们通常无法以合适的位置和形式保住可复用的判断结构。

现有基础设施更擅长记录的是：

- 代码、配置、拉取请求（PR）和议题（Issue），它们记录了最后改了什么
- 文档和会议纪要，它们记录了讨论过什么
- 追踪（tracing）、日志和可观测性数据，它们记录了执行了什么
- 知识库、维基或客户关系管理（CRM）系统，它们记录了被显式存下来的内容

而 OpenPrecedent 想保住的是另一种单位：

- 为什么默认路径在这个情境里不再成立
- 是哪些约束迫使团队或 Agent 改走别的路径
- 哪些被否掉的选项对后续判断仍然重要

这类单位之所以难以被现有系统保住，是因为它通常同时具备几个特征：

- 它分散在多个表面上
- 它只在最终产物里留下了部分痕迹
- 它足够情境化，不值得单独写成长文档
- 它在决策发生当下最清晰，事后就迅速模糊

人工登记系统也是一个自然替代方案，但它通常不适合做高价值例外决策的主要捕获机制。
主要原因是：

- 额外流程成本恰好出现在团队最忙、最赶、最在处理例外的时候
- 人在当下往往并不知道哪一次“看似临时”的选择，后面会变成关键历史决策
- 人工总结很容易和真实执行路径、证据链条、决策时刻脱节

人工系统当然仍然有价值，但它更适合作为补充层，而不是主要入口。

## 为什么 Agent 时代改变了需求强度

OpenPrecedent 在 Agent 时代变得更必要，最核心的原因是：决策从相对低频的组织活动，变成了高频的执行活动。

相比以人为主的工作方式，Agent 辅助工作带来了几个根本变化：

- 单个任务内部的决策密度显著上升
- 许多决策不再只是人和人之间讨论出来，而是在人与 Agent 的交互中共同形成
- 有些中间决策由 Agent 在执行过程中完成，人并不会逐个过目
- 默认路径更危险，因为 Agent 天然更容易滑向通用和主流解
- 执行速度更快，人类回忆更难在正确时刻插入
- 被捕获的先例可以直接被下一次执行消费，而不只是供后人阅读

这会带来一种质变。
在以人为主的时代，很多判断还可以长期保留在少数专家脑中，因为后续的人通常还能在行动前通过回忆、沟通和追问补回来。
但在 Agent 时代，决策数量和速度都抬高了，这种纯靠人的补洞机制会越来越脆弱，也越来越贵。

所以沿执行路径捕获，重要的不是“多保留一点历史”，而是让历史判断在下一次相似决策发生时仍然可用，而不是只能在事后解释过去。

## 先例是什么，不是什么

一个关键风险是，把被记录的决策误当成可以永久复用的正确答案。
如果系统这样工作，它会很危险。

因为一个先例可能是：

- 在当时就做错了
- 只在非常局部的约束下成立
- 后来随着产品、客户、架构或组织变化而失效

所以 OpenPrecedent 不应该把先例当成规则或真值，而应该把它当成带上下文的历史判断样本。

这三类风险其实不是一回事：

- 错误决策，意味着原始判断本身就是错的
- 局部正确决策，意味着它只在非常窄的约束下成立
- 后来失效决策，意味着它可能当时是对的，但已经随着环境变化过期

它们带来的污染也不同：

- 错误先例会把历史错误放大
- 局部正确先例会被过度泛化到不该适用的地方
- 已经过期的先例会把未来工作锚定在不再成立的前提上

所以真正的危险不只是数据不干净，而是历史判断被错当成持续有效的权威。
一旦系统把先例呈现得像永久有效的规范，未来 Agent 就更容易停止追问当前约束是否真的匹配。

这意味着，未来的 Agent 不能只继承决策内容本身，还应该同时继承：

- 当时成立的约束条件
- 被考虑过或被否掉的替代方案
- 当时可见的证据
- 后来的结果或复盘评价（如果已有）
- 这个决策是否只是临时性的、例外性的，或者后来已经被替换

产品最有价值的状态，不是让未来执行默认相信“以前这么做过，所以现在也该这么做”，而是帮助它继续追问“这个历史判断，在当前情境下还成立吗？”

这也意味着产品边界需要很清楚：

- OpenPrecedent 应该浮现候选先例，而不是静默强制历史模仿
- 检索质量依赖的不只是主题或措辞相似度，更依赖约束和适用性的匹配
- 如果系统能够保留后来的修正、替代和失效信息，它会比只保留原始决策本身更有价值

因此，污染控制更合理的方式不是一步到位做成重治理系统，而是分层处理。
更现实的方式是：

- 在决策发生时，先把与之相邻的约束和上下文一起保留下来
- 等到后续结果、替代、失效信息逐渐明确后，再允许补充增强
- 在检索和使用时，把它当成需要重新判断的历史样本，而不是自动服从的规则

## 应该记录什么

最有价值的先例，通常不是那些已经被代码、文档或通用最佳实践充分表达出来的标准选择。
最有价值的先例，反而是那些偏离默认路径、后来又很难重新推回来的决策。

当前的工作假设是，应优先沉淀这几类决策：

- 脱离最初设计路线的调整
- 缺陷修复过程中的权衡与取舍
- 客户特殊要求带来的例外选择
- 某些版本因为时间压力而做出的裁剪决定
- 没有完整停留在文档和代码里的架构选型细节
- 缺陷修复、事故处理过程中采用的临时方案与过渡方案

这些决策特别值得沉淀，是因为它们往往同时具备几个特征：

- 重新发现它们的代价很高
- 它们在代码或最终文档中的表达非常弱
- 它们会深刻影响后续维护和演进
- 一旦人员流动或角色变化，就特别容易丢失

这套判断并不只适用于开发过程。
如果进一步泛化，类似逻辑也适用于：

- 改变路线或范围的重要产品规划决策
- 影响承诺边界的重要销售或客户交付决策

更一般化的一句话是：

OpenPrecedent 最应该优先沉淀的，是那些最难从最终产物里反推回来、却又最影响后续判断的决策。

如果收紧到第一阶段，优先级还可以更明确：

OpenPrecedent 最先应该沉淀的，是那些在真实约束下被迫偏离默认路径的例外型决策。尤其是本来按主流方案就会走通、但因为客户要求、历史包袱、缺陷修复权衡、版本裁剪压力、未文档化的架构细节，或临时恢复需要，而不得不改走另一条路的地方。

如果再进一步收紧到第一批真实样本，最合适的切口是：

OpenPrecedent 最先应抓取那些因为客户特异性约束或历史兼容性约束，而不能采用默认实现方案的开发决策。

这类样本特别适合作为首批真实捕获对象，因为它们同时具备：

- 一旦重复犯错，代价通常很高
- 后续 Agent 很容易自然滑回主流答案
- 单靠代码结果很难反推出当时为什么不能按常规做
- 往往天然分布在议题、拉取请求、运行时记录、支持语境和专家解释里，比较容易形成首批闭环

如果给第一批样本再排一个顺序，可以优先考虑：

- 覆盖标准产品路径的客户特例
- 覆盖推荐工程路径的兼容性或历史包袱约束
- 在局部压力下被明确接受的临时方案或过渡方案
- 改变了“当前版本做到什么算够用”的版本裁剪决定

## 应该怎么捕获

在系统落地初期，如果假设可以全自动捕获全部关键决策，通常是不现实的，因为大量重要决策仍然发生在会议、邮件、聊天系统和临时口头讨论里。

更可行的早期模型应该是混合捕获：

- 对 Agent 可见的执行路径进行自动或近自动捕获
- 对人和 Agent 的交互过程做结构化捕获
- 对 Agent 无法直接感知的人际沟通，提供低摩擦的人工补充入口

也就是说，早期目标不是全知全能，而是先抓住那些最高价值、最难重建的决策，并保留足够上下文，让后续 Agent 和操作者可以重新利用。

如果再把边界收紧一点，可以分成三层：

第一层，是应该优先沿执行路径自动抓取的决策，例如：

- Agent 提出多个方案后，人从中选择一个
- Agent 本来会走默认路径，但因为成本、权限、兼容性或客户约束而改走另一条路
- Agent 在执行过程中做出的、虽未被人逐条审阅但实质影响结果的中间判断

第二层，是必须允许人补录的高价值决策，例如：

- 在会议里拍板的客户特例
- 在邮件或聊天中澄清的交付边界
- 专家解释某个主流修复方案为什么在当前仓库里并不安全

第三层，是只需要关联证据材料、不必强行全部结构化的上下文，例如：

- 长会议纪要
- 聊天记录串
- 设计稿、录屏、需求稿
- 长邮件线程

这类材料的重要性通常在于证明和补充，而不在于全部拆成字段。
如果系统试图把它们全部结构化，前期成本会过高，也会引入大量噪音。

## 候选待完成任务

OpenPrecedent 不应该把自己理解成单独售卖回放、存储或检索。
用户真正雇用它，是为了在下一步关键动作前降低决策风险，避免重复已经被否掉的推理，浮现仓库特异性和客户特异性的例外条件，并缩短“我们以前做过类似事情”和“这次还应该继承哪部分判断”之间的距离。

在进一步收紧待完成任务之前，还需要先把客户定义对。
客户不能被过于宽泛地描述成“开发者”。
这就像奶昔案例里只说“购买者是某个年龄段的通勤者”一样，信息还不够有用。

更有效的工作假设应该分成三层：

- 直接使用者，是那些需要在仓库特异性或客户特异性约束下反复做项目决策的工程师、技术负责人、交付负责人和 Agent 操作者
- 组织层客户，是那些系统会持续演进、同时不断积累特例、历史包袱和隐性例外的软件团队与产品组织
- 经济购买者，通常是要为维护成本、返工成本和专家依赖负责的研发负责人、平台负责人、AI 工作流负责人、交付负责人或产品线负责人

这些客户真正共享的，不是岗位名称，而是在真实历史约束下反复做新决策的需要。
他们的问题不是“缺少更多信息”，而是缺少能在下一次决策时被重新调用的历史判断。

### 候选主任务：新决策时的判断继承

当用户或 Agent 需要在真实项目里做出一个有分量的新决策时，他们需要一份简短但高密度的视图，告诉他们过去类似情境已经证明了哪些任务框定、约束条件、代价权衡、权限边界和成功标准是重要的。

用待完成任务句式表达就是：

当我需要在真实项目里做一个带有真实权衡的新决策，例如选方案、算成本、平衡速度、维护负担、审批代价或客户特异性例外时，我需要一份来自相似历史案例的判断结构简报，这样我就能用更少猜测、更少返工和更低风险推进当前工作，而不是重复走已经被否掉的路径。

规划、写入、恢复和审批边界这些时刻仍然是重要触发点，但它们不是全部。
更深一层的任务，是在真实项目权衡下支持新的决策本身，而不只是守护执行节点。

这是当前最强的产品切入点候选，因为它发生在用户最愿意改变行为、也最愿意在制造新的下游代价之前参考历史判断的时刻。

### 次任务：事后解释决策

当一次 Agent 执行结果看起来可疑时，用户需要快速还原某个决策为什么发生，以及判断到底错在什么地方。

这个任务很有价值，但单靠它不太适合作为最强切入点，因为许多可观测性工具已经部分覆盖了相邻的事后复盘需求。

### 长周期任务：持续沉淀可复用先例

随着时间推移，团队会希望把项目特异性和客户特异性的判断模式沉淀成可复用的先例，而不是散落在部落记忆里。
这里的关键不是一步到位替代专家判断，而是在真实工作过程中先辅助专业人员与 Agent 协作，再逐步把可复用的判断沉淀出来，降低对脆弱人工回忆的依赖。

这个任务同样重要，但它更像长期复利收益，而不是最窄、最容易打穿的首个产品切口。

### 客户期望获得的结果

客户雇用 OpenPrecedent，并不是为了拥有一个决策档案馆。
他们真正期望获得的结果更偏执行层面：

- 在做新决策前找回仍然有效的约束、例外和曾被否掉的路径
- 减少“看起来合理、其实错误”的默认解
- 减少反复去问同一个专家、让对方重新回忆历史的成本
- 让项目特异性和客户特异性的判断，在团队变化、角色变化和 Agent 使用增加时仍然不轻易丢失

一句话说，客户雇用 OpenPrecedent，是为了在历史约束下减少决策错误，而不只是为了提升文档完整度。

## 替代方案与采用门槛

用户今天已经有很多方式可以完成这项任务的一部分。
这些替代方案才是真正的竞争对手。

主要替代方案包括：

- 不消费，直接让 Agent 继续往下走
- 人的回忆与专家咨询
- 代码搜索与仓库历史
- 可观测性与追踪工具
- 记忆、检索增强生成（RAG）与笔记检索系统
- 流程模板、政策文档与检查清单

其中最强的替代方案通常是人的回忆与专家咨询，因为人比当前工具更擅长重建意图和例外。
它同时也是 OpenPrecedent 最需要长期替代掉的替代方案。这个问题是结构性的：

- 关键决策信息往往只活在专家记忆里
- 获取专家回忆既慢又贵
- 人员流失和组织变更会让这套记忆基础越来越脆弱
- 项目越长期演进，越难持续维护这种必须问对人的依赖

OpenPrecedent 不应该假设自己能一步替代专家判断。
更合理的路径是：把专家与 Agent 的反复交互逐步沉淀成显式先例，让系统随着语料积累而降低对专家回忆的依赖。

只有当 OpenPrecedent 在下列几个方面比替代方案组合做得更好，用户才更可能真正采用它：

- 比翻阅 PR、文档、聊天和笔记更快地返回相关判断
- 返回的是判断结构，而不是一堆相关文本
- 出现在规划、写入或失败恢复这些关键时刻，而不是只在事后出现
- 能保住仓库特异性和客户特异性例外，而不是把一切压平成通用最佳实践
- 能解释为什么某条先例相关，使结果具备可审计性和可信度

用户仍然可能在下面这些情况下选择替代方案：

- 任务太小、风险太低，不值得多走一步
- 几乎没有值得参考的历史
- 产品返回了太多原始历史，蒸馏过的指导太少
- 最关键的决策信息仍然在已捕获语料之外
- 用户更信任人类专家，而不是检索到的先例
- 产品看起来像另一个可观测性面板，而不是行动时辅助工具

这也意味着，把回放做得更好本身很可能不够。
如果 OpenPrecedent 无法打赢搜索加人工回忆这个组合流程，它就很难变成稳定习惯。

这也可以把产品价值再简化一层。
OpenPrecedent 的核心价值，不是因为它多记录了一份历史，而是因为它有机会减少那些发生在例外约束下的错误默认决策。

这种价值通常表现为：

- 因误走看似合理、其实不成立的主流路径而产生的返工更少
- 已经被否掉的推理不必反复重新发现一次
- 对少数专家即时回忆的运行性依赖更低
- 仓库特异性和客户特异性例外被更稳定地处理

## 如何判断 OpenPrecedent 真的有用

光是把决策记录下来，还不够证明产品有价值。
真正有价值的先例层，必须改变后续判断，而不只是多存了一份历史。

因此，验证标准最好分层看：

### 第一层：捕获成功

系统能否稳定地产出来自真实工作的候选先例。
这一层只能证明语料开始存在。

### 第二层：检索成功

当后续出现相似情境时，系统能否在正确时刻找回相关先例。
这一层证明历史能在需要时被取回。

### 第三层：判断改变成功

检索出的先例是否真的改变了 Agent 或操作者下一步的选择。
较强的信号包括：

- Agent 因看到历史例外而放弃一个很诱人的默认解
- Agent 比原来更早地主动提出澄清或审批需求
- 用户或 Agent 更倾向于选择符合仓库特异性或客户特异性约束的路径，而不是通用最佳实践
- 已经被否掉的推理，不再需要重新走一遍再被否掉一次

### 第四层：结果改变成功

这种判断改变是否带来了真实的下游收益。
例如：

- 返工减少
- 看起来合理但其实错误的默认实现减少
- 对专家即时回忆的依赖下降
- 临时修复方案不再被误当成长期设计
- 客户特异性和仓库特异性例外被更稳定地处理

所以，OpenPrecedent 的成功门槛不应该是记录了多少历史，而应该是未来的执行是否因此需要更少猜测、更少重新发现，也更少依赖去问对的人。

## 开源与商业化边界

OpenPrecedent 不应该假设，只要是开源就天然成立，也不应该假设，只要做成闭源包装就天然能卖。
更关键的问题是：哪些部分适合成为开源的先例层基础，哪些部分会构成企业客户愿意付费的理由。

开源基础更有优势的部分包括：

- 本地优先的先例模型
- 透明可审计的捕获、回放、解释与检索原语
- 面向开发者的信任、可扩展性和生态采用
- 供团队自行试验、自托管和扩展的基础底座

闭源或私有云产品只有在解决了客户很难自行拼装和维护的问题时，才真正有防御力，例如：

- 企业部署、多租户、安全和合规要求
- 仓库系统、聊天系统、会议材料、支持系统与 Agent 运行时之间的连接器
- 跨团队的先例质量、失效、访问边界治理
- 落地实施、行业化推广和结果责任承担

这意味着，商业版不能把“闭源”本身当成优势。
真正的优势应该来自更深的集成、更强的企业适配，以及对落地结果负责。

这也解释了为什么有些客户即使知道有开源版本，仍然会选择商业化产品：

- 他们想比自研拼装更快落地
- 他们需要供应商承担支持和集成风险
- 他们有私有云、合规或数据边界约束
- 他们需要先例捕获跨越更多企业系统，而内部平台团队很难长期维护这套整合

因此，这门生意更像是在卖组织级能力，而不只是卖一个窄意义上的开发者席位工具。
更合理的收入抓手可能包括：

- 企业版或私有云部署订阅
- 按被管理的项目、Agent 工作流或组织范围定价，而不只是按单个席位收费
- 面向企业落地和系统集成的实施服务或交付包

这里还需要一个战略提醒：产品可以从开发者与 Agent 工作流切入，但不能把自己困死在“只是另一个开发者工具”里。
更深的机会，是从软件交付中的判断继承切入，逐步延伸到产品、交付和客户例外等相邻决策场景。

## 当前最小可行产品与未来目标

当前 OpenPrecedent 的最强能力主要在于：

- 捕获本地案例历史
- 抽取决策记录
- 回放并解释一个案例
- 从已有历史中检索语义上相关的先例

这些能力非常重要，但更适合被理解为证明这条产品路径可行的基础设施底座。
它同时也是先在一个高信号、较窄的环境里证明这条闭环，而不是试图一次覆盖所有 Agent 领域。

更强的未来产品目标应该更窄、更明确：

- 在下一次关键动作前浮现决策谱系
- 整合不止来自原始运行时转录的决策相关信号
- 把客户特异性和仓库特异性例外沉淀成一等先例
- 帮助 Agent 继承已经被证明有效的判断结构，而不是只检索邻近文本或操作相似性
- 把专业人员与 Agent 协作过程中形成的经验持续沉淀成可复用先例，让组织获得复利，而不是每次都退回部落记忆

未来产品不应该滑向通用图谱、通用记忆平台或通用追踪查看器。
它真正的独特价值，是在真实项目约束下完成决策继承。

但从长期看，这个价值不应该被锁死在编码 Agent 上。
真正可泛化的层，不是“软件项目历史”本身，而是面向 Agent 辅助执行的情境化决策先例层。
如果这套模型成立，未来应该是同一套核心对象模型，叠加不同领域的连接器、决策分类法和评估逻辑，逐步支持更多 Agent 场景。

上面这些设计问题其实是相互锁定的，而不是彼此独立的话题：

- 污染控制决定了哪类历史判断可以被安全复用
- 混合捕获决定了这些判断如何进入系统
- 第一批样本范围决定了产品从哪里开始最容易形成高价值闭环
- 有用性的判断标准决定了系统是在改变后续决策，还是只是更好地归档历史

## 开放问题

- OpenPrecedent 未来应该直接吸收多少对话外上下文，又应该间接引用多少？
- 哪一个高频触发时刻最值得成为最小切入点：规划前、写入前、失败恢复后、审批边界处，还是更广义的方案权衡节点？
- 产品应如何区分稳定先例与一次性的本地例外？
- 在不滑向通用客户关系管理（CRM）或通用知识库的前提下，客户特异性例外应该如何建模？
- 什么样的证据才能证明“新决策时的决策谱系”真正改变了后续决策，而不只是让事后解释更顺？
- 哪些专家判断最有可能被优先外化成先例，哪些判断又会在更长时间内继续依赖专家本人？
