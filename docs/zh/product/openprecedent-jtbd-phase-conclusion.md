# OpenPrecedent 待完成任务阶段性结论

日期：2026-03-26
状态：阶段性结论
英文对应稿：[docs/product/openprecedent-jtbd-phase-conclusion.md](/root/.config/superpowers/worktrees/openprecedent/codex-jtbd-product-wedge/docs/product/openprecedent-jtbd-phase-conclusion.md)

## 为什么需要这份结论文档

这份文档用于收束 OpenPrecedent 当前这轮待完成任务（Jobs To Be Done，JTBD）讨论。
之前那份工作文档更多承担了探索和迭代的作用。
而这份文档的目标，是把目前已经相对稳定、足以指导后续产品工作的判断整理成阶段性结论。

这里采用的参照框架，来自《与运气竞争》。
最重要的问题不是简单地问“用户是谁”，而是继续追问三件事：

- 用户会在什么情境下雇用这个产品
- 用户真正想推进的进展是什么
- 用户今天已经在用什么替代方案完成这件事

按照这个框架，这份文档总结当前已经回答的问题、尚未完全回答的问题，以及当前最小可行产品（MVP）与这些判断之间的对齐和差异。

## 核心结论

OpenPrecedent 不应该被主要理解为一个决策档案系统、Agent 日志查看器，或者一个泛化的记忆存储系统。
当前最强的结论是：当一个新的关键决策即将发生时，用户雇用 OpenPrecedent，是为了找回过去仍然相关的历史判断。

产品最有价值的地方，不是把更多历史存下来，而是在新的决策现场，把仍然重要的约束、例外、被否掉的路径和历史权衡重新带回来。
它的直接价值，不是“信息更多”，而是减少错误默认决策。

一句话说：

OpenPrecedent 应该成为一层面向后续执行的决策先例层，让真实约束下形成过的历史判断可以被再次继承。

## 这轮讨论已经回答了什么

### 用户到底在“买”什么

用户不是为了拥有一个“决策记录系统”才需要 OpenPrecedent。
他们也不是为了拥有一个“Agent 日志查看器”才需要它。

他们真正雇用 OpenPrecedent 的时刻，是在一个新的、有后果的决策即将发生时，他们不想再从零重新猜过去为什么那样判断。
他们真正想取回的，也不是原始历史本身，而是那些今天仍然关键的历史部分：

- 哪些约束当时起了决定作用
- 哪些例外覆盖了默认路径
- 哪些方案被认真评估过并最终否掉
- 哪些审批边界和责任边界当时重要
- 哪些历史判断今天仍然应该影响当前决策

这是这轮讨论最重要的收敛结论。

### 真正的竞争对手是谁

OpenPrecedent 的竞争对手，不只是可观测性、追踪或记忆系统。
真正的竞争对手，是用户今天已经在使用的一整套组合替代方案：

- 不消费，直接让 Agent 继续往下走
- 人的回忆与专家咨询
- 代码搜索与仓库历史
- 文档、会议纪要、Slack 线程、聊天系统和邮件
- 可观测性与追踪工具
- 记忆、检索增强生成（RAG）与笔记检索系统
- 流程模板、政策文档和检查清单

其中最强的替代方案，通常并不是另一个软件产品，而是人的回忆与专家咨询。

这点很重要，因为它直接决定了采用门槛。
如果 OpenPrecedent 无法打赢“搜索加专家重建”这个组合流程，它就很难变成稳定习惯。

### 为什么现有方式不够

问题不在于现有系统“什么都没记录”。
问题在于，它们通常无法把未来决策真正需要的判断结构稳定保住并重新利用。

现有方式往往保留了很多相邻信息，但没有保住最可复用的语义单位。
尤其容易丢失的是：

- 为什么默认路径不再成立
- 哪些约束强到足以迫使偏离默认路径
- 哪些选项被认真否掉，以及为什么
- 哪些例外只在局部成立
- 哪些决策后来已经失效、被替代，或不该继续信任

所以真正的缺口不是“没有记录”，而是未来决策真正需要的、带上下文边界的历史判断，没有被以可复用方式保留下来。

### 为什么 Agent 时代让这个问题更严重

这轮讨论已经较完整地回答了：Agent 时代并不是凭空创造了决策，而是改变了决策的密度、可见性和下游影响。

Agent 时代让这个问题更重要，是因为：

- 单个工作单元里的决策密度更高
- 很多决策发生在人与 Agent 的交互中，而不再只是人和人之间
- 有些中间决策由 Agent 完成，人不会逐条审阅
- Agent 更容易滑向主流解或默认解
- 未来执行者也越来越是 Agent，而不是总能临场补洞的人

这让先例的意义发生了变化。
先例不再只是给人事后复盘，而是要被下一次执行直接消费。

### 应该优先记录什么

这轮讨论也已经比较明确地收敛了记录优先级。

OpenPrecedent 不应该从泛泛记录“重要决策”开始。
更合理的优先级，是先抓那些偏离默认路径、很难从最终产物反推、但又会强烈影响后续维护与判断的例外型决策。

目前识别出的高价值类型包括：

- 客户特异性例外
- 历史兼容性约束
- 缺陷修复中的权衡
- 临时方案与过渡方案
- 因时间压力导致的版本裁剪
- 没有完整进入代码或文档的架构细节

比这些例子本身更重要的是它们背后的共同模式：

最适合先沉淀的先例，是那些本来会走默认路径、但因为真实约束而不得不改走另一条路的决策。

### OpenPrecedent 的产品本体是什么

这轮还有一个重要纠偏：OpenPrecedent 不能被简单描述成开发者项目先例工具。

它的长期产品类别应该更宽：

OpenPrecedent 应该被理解成一层面向 Agent 辅助执行的通用决策先例基础设施。

这类基础设施理论上可以覆盖：

- 编码与软件交付 Agent
- 销售 Agent
- 运营与组织流程 Agent
- 个人助理 Agent
- 工业或运营 Agent

但与此同时，当前高信号的验证环境仍然是窄的：

- 当前首个切入点仍然是编码 Agent 与软件交付

所以最准确的表达不是在“开发者工具”和“通用平台”之间二选一，而是同时成立两层：

- 产品类别是通用的
- 首个切入点是具体的

## 对几个核心问题的直接回答

### 客户是谁

客户不能被压缩成一个笼统的“开发者”标签。
更准确的客户模型应该分成三层。

直接使用者，是那些高频与 Agent 协作、并需要在项目特异性约束下持续做决策的人。
这包括工程师、技术负责人、交付负责人，以及其他 Agent 辅助流程中的操作者。

组织层客户，是那些系统会长期演进、同时不断积累特例、历史包袱和隐性判断的软件团队或产品组织。

经济购买者，则是那些要为下游代价负责的人，通常承担维护负担、返工成本、交付风险，或组织对专家记忆依赖的责任。

### 他们雇用 OpenPrecedent 为了什么

他们不是为了拥有一个档案库才雇用 OpenPrecedent。
他们雇用它，是为了在新的关键决策发生时，不必重新猜过去为什么那样判断。

可以把这个任务表达成：

当一个新的关键决策必须在真实权衡下做出时，我希望系统能从相似历史案例里恢复仍然相关的历史判断，这样我就可以用更少猜测、更低风险推进当前工作，而不是重复走错误的、或者早已被否掉的默认路径。

### 他们期望什么结果

他们期望的不是抽象意义上的“知识更多”。
他们真正期望的是更偏执行层面的结果：

- 错误默认路径更少
- 返工更少
- 对在对的时间问对的人这件事依赖更少
- 团队变化时，特异性判断不容易丢
- Agent 在下一次行动前更像“继承过经验”，而不是“重新靠平均值猜”

### OpenPrecedent 的核心价值是什么

如果把这轮讨论压缩成一句对外价值表达，我认为最短、也最有力的一句是：

OpenPrecedent 减少错误默认决策。

这句话刻意比产品内部机制更简单，但它最接近外部能感知到的核心价值。

### 为什么不是人工登记系统

高价值决策往往发生在最忙、最乱、最临时、最像例外的时刻。
这也正是人工登记最不可靠的时候。

人工记录仍然可以作为补充手段。
但它不适合作为主捕获机制，因为最重要的决策恰恰往往最不可能在发生当时被结构化、主动地写下来。

### precedent 会不会污染未来决策

会。
这一轮已经非常明确地把 precedent 定义成一种可能危险的对象，如果它被误当成永远正确的真理。

precedent 可能是：

- 在当时就是错的
- 只在一个很窄的局部条件下成立
- 后来因为产品、客户、架构、组织或政策变化而失效

所以 precedent 必须被理解为带上下文边界的历史判断样本。
这也是为什么这轮结论会强调：

- applicability（适用边界）
- invalidation / supersession（失效与替代）
- candidate precedent（候选先例），而不是 silent enforcement（静默强制执行）

### 为什么产品既要通用，又不能太发散

当前的收敛判断是：

- 愿景必须通用
- 验证必须具体

因此，OpenPrecedent 应该被描述成通用的 Agent 决策先例基础设施，但当前首个切入点仍然是编码 Agent 与软件交付场景。

这个分层是必要的。
如果没有通用层，产品会塌缩成一个窄场景工具。
如果没有具体切入点，产品又会变得过于抽象而无法验证。

## 还有哪些问题没有完全打透

虽然这轮讨论已经足够收敛，但仍有一些问题只是“方向清楚”，还没有进入“定义完成”的状态。

### 最小目标客户还没有最终锁死

文档已经不再把“开发者”当成最终答案，但第一批最应该买单、最强烈感受到痛点的团队类型，还没有最后锁死。
下一轮仍然需要定义最窄的首批购买团队。

### 核心字段与增强字段还没有最终切开

文档已经记录了更清楚的语义内核，包括 case、decision、precedent、适用边界等候选字段。
但它还没有彻底区分：

- 哪些字段是 MVP 语义内核必须有的
- 哪些字段可以保持可选，作为后续增强

### precedent 在运行时怎样被谨慎消费，还没有最终定义

分析已经很明确地指出 precedent 不能被盲目服从。
但产品还没有完全定义在运行时应该何时：

- 提示一条先例
- 要求澄清
- 阻断默认动作
- 升级给人处理
- 或只在事后补充解释

### 商业化边界虽有方向，但还没有形成最终产品包装

开源底座与私有云企业版的方向已经清楚了。
但这些判断还没有完全压缩成真正的产品 packaging。
也就是说，边界已经清楚，但包装仍未最终定义。

这些问题已经不再属于“方向不清”。
它们属于下一阶段的产品定义问题。

## 与当前 MVP 的对齐情况

当前的 MVP 与这份阶段性结论，在方向上其实相当一致。

### MVP 已经对齐的部分

MVP 已经和这份结论共享同一条核心结构链路。
它已经对齐的地方包括：

- 它已经围绕 case、event、decision、precedent 这条链路组织能力
- 它并没有把自己定义成通用图谱、通用记忆平台或通用追踪查看器
- 它已经把 replay、explanation 和 precedent retrieval 放在核心能力里
- 它已经在验证“从已有历史里检索相似 precedent”这件事
- 它已经强调 local-first、单 Agent、可具体验证，而不是抽象平台叙事
- 它已经遵循了正确的验证顺序：先把一个具体闭环证明出来，再谈扩展

因此，MVP 不应该被评价成方向错误。
它已经抓住了更大产品论点的骨架。

### MVP 仍然存在的差异

当前 MVP 和这份结论的主要差异，不在于它走错了方向，而在于它当前所处的抽象层级，还低于这份结论所定义的产品目标。

当前 MVP 仍然更像：

- 一个面向已有历史的捕获、回放、抽取和检索底座

而这份阶段性结论描述的更强目标是：

- 在下一次关键动作发生前，提供决策时刻的判断继承

主要差异包括：

- MVP 目前仍然更偏向解释过去发生了什么，而阶段性结论更强调在新决策发生前帮助下一步判断
- MVP 主要验证的是本地单 Agent 的编码场景，而阶段性结论定义的是更宽的 Agent 辅助执行产品类别
- MVP 对会议、聊天系统、邮件、专家补录等“路径之外的人类上下文”覆盖仍然较弱，而阶段性结论已经把混合捕获当作必要条件
- MVP 还没有把 applicability、invalidation、supersession、temporality 提升到足够中心的位置，而这份结论认为它们是防止 precedent 污染未来决策的关键
- MVP 目前更偏 semantic retrieval 加 replay，而更强的产品论点要求加入约束匹配、例外识别和历史有效性判断
- MVP 还没有把整体架构明确表达成 core precedent layer、domain adaptation layer 和 runtime usage layer

因此，最准确的一句话总结是：

MVP 已经证明 precedent loop 可以跑起来，但它还没有完全长成这份阶段性结论所定义的“通用 Agent 决策先例基础设施”。

## 阶段性总结

如果借用《与运气竞争》的语言来评价这轮收敛，可以说现在的讨论已经比最开始清楚很多。
OpenPrecedent 已经不再被理解成一个“功能很多、但定义松散”的系统。
这轮分析已经对下面这些问题给出了比较连贯的回答：

- 用户到底雇用它完成什么任务
- 它真正要打赢的是哪组替代方案
- 为什么这个任务在 Agent 时代更重要
- 什么类型的决策最值得优先沉淀
- 它的长期产品类别是什么
- 它的首个验证切入点又是什么

因此，当前 MVP 和这份阶段性结论是同方向、不同阶段的关系：

- MVP 证明这条闭环可能成立
- 这份结论定义这条闭环最终应该长成什么
