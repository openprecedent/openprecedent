## 一句话定义

围绕单机 OpenClaw 的真实协作过程，验证 Context Graph 最小闭环是否成立：把一次用户任务中的 agent 轨迹、关键决策、解释证据和历史先例沉淀为可回放、可审计、可复用的 case。

## 1. 背景

终局产品文档已经明确，Context Graph 的长期方向不是通用图数据库，也不是通用 observability，而是面向 Agent 决策对象的记录、审计和复用基础设施。

但在进入更大范围的企业级场景前，必须先回答一个更基础的问题：

- 如果只围绕一个真实运行中的 agent
- 如果只围绕一个人和这个 agent 的高频协作
- 如果只围绕一条完整任务链路

系统能否稳定回答以下问题：

- agent 做了什么
- 为什么这么做
- 哪一步做错了
- 下次能否复用这次经验

因此，MVP 不以多租户平台为起点，而以“本机 OpenClaw + 用户真实任务”作为最小验证单元。

## 2. 核心判断

MVP 阶段要验证的不是“大图谱价值”，而是三个最小价值是否同时成立：

1. 一次任务的完整轨迹能被可靠记录
2. 关键决策能被提炼并解释
3. 历史 case 能被检索并形成先例价值

如果这三个价值不能在单机 OpenClaw 场景里成立，那么后续扩展到销售数字人、编码 agent 或技术支持 agent 时，复杂度只会更高，不会更低。

## 3. 核心用户与场景

### 3.1 核心用户

- 与本机 OpenClaw 高频协作的深度用户
- 需要复盘 agent 行为的产品、工程或运营人员
- 需要审计 agent 决策依据的负责人

### 3.2 核心场景

第一阶段只验证一个最小场景：

- 用户向 OpenClaw 提出一项真实任务
- OpenClaw 通过对话、搜索、读取文件、执行命令、调用工具或修改文件推进任务
- 系统自动记录全过程
- 用户事后查看完整轨迹与关键决策
- 系统返回相似历史 case 供用户复用

### 3.3 MVP 中的 case 定义

一个 case 指一次明确任务从开始到结束的完整任务生命周期，通常包括：

- 用户发起任务
- agent 多轮响应
- 工具或命令执行
- 关键判断
- 人工确认或覆盖
- 最终完成、失败或暂停

## 4. 目标

### 4.1 总目标

验证 Context Graph 在单机 OpenClaw 场景中，是否能够把“一次任务中的决策过程”沉淀成可回放、可解释、可检索的结构化对象。

### 4.2 MVP 目标

- 自动采集一次 case 的完整事件轨迹
- 从轨迹中抽取高价值决策节点
- 支持按 case 回放原始过程与决策过程
- 为关键决策生成证据绑定的解释
- 支持基于历史 case 的最小先例检索

### 4.3 非目标

第一阶段明确不做：

- 多 agent 协同图谱
- 通用企业级多租户平台
- 全量知识图谱建模
- 自动决策优化引擎
- 大规模 workflow 编排
- 通用图数据库能力验证

## 5. 产品主张

MVP 要交付的不是一套抽象“图能力”，而是三项直接可感知的用户价值：

- 事后能看懂：知道 agent 完整做了什么
- 出问题能追责：知道关键判断为什么发生、错在何处
- 下次能复用：知道历史类似 case 如何处理

## 6. 核心流程

### 6.1 流程一：启动 case

1. 用户向 OpenClaw 发起任务
2. 系统创建新的 case
3. 系统开始记录该 case 的全部事件

### 6.2 流程二：执行与采集

1. OpenClaw 与用户持续对话
2. OpenClaw 读取文件、调用工具、执行命令或写入文件
3. 系统按统一 schema 记录原始事件
4. 系统在后台识别关键决策候选

### 6.3 流程三：结束与结构化

1. case 完成、失败或暂停
2. 系统生成 case 摘要
3. 系统抽取关键决策链
4. 系统为决策链绑定证据和解释

### 6.4 流程四：回放与解释

1. 用户打开某个历史 case
2. 系统展示原始时间线
3. 系统展示关键决策链
4. 用户点击节点查看解释、证据和结果

### 6.5 流程五：先例检索

1. 用户查看当前 case 或当前决策节点
2. 系统返回历史相似 case
3. 系统展示相似原因、差异点和历史处理结果

## 7. 范围

### 7.1 In Scope

- 单机 OpenClaw 单实例场景
- 单 case 维度的轨迹采集
- 对话、工具调用、命令执行、文件操作的统一事件记录
- 关键决策节点抽取
- 单 case 回放
- 决策解释
- 历史相似 case 检索

### 7.2 Out of Scope

- 组织级权限治理
- 复杂审批流
- 第三方系统深度集成矩阵
- 多实例统一运营看板
- 跨 case 复杂图分析
- 自适应策略优化

## 8. 功能需求

### 8.1 模块一：Case 与事件采集

系统必须支持：

1. 为每次用户任务创建唯一 case。
2. 记录用户消息与 agent 消息。
3. 记录模型调用输入输出的元数据。
4. 记录工具调用、命令执行、文件读写、错误和重试。
5. 保留事件顺序、时间戳和上下游关联关系。

### 8.2 模块二：决策节点抽取

系统必须支持：

1. 从事件流中抽取高价值决策节点，而不是把所有事件都视为决策。
2. 至少识别以下决策类型：
   - 澄清问题
   - 制定计划
   - 选择工具
   - 执行修改
   - 恢复与重试
   - 完成与交付
3. 为每个决策节点关联其证据事件。
4. 标记人工确认、人工覆盖或例外情况。

### 8.3 模块三：Case 回放

系统必须支持：

1. 按时间顺序回放单个 case 的完整事件流。
2. 展示关键决策链。
3. 在同一个 case 中区分“原始轨迹”和“决策对象”两层视图。
4. 查看每个事件或决策节点的上下文详情。

### 8.4 模块四：决策解释

系统必须支持：

1. 对每个关键决策回答：
   - 当时要解决什么问题
   - 看到了哪些证据
   - 受哪些约束影响
   - 为什么采取当前动作
2. 解释必须绑定具体事件或片段，而不是纯生成式总结。
3. 解释中必须标记人工确认或人工覆盖是否参与。

### 8.5 模块五：先例检索

系统必须支持：

1. 为每个 case 生成可检索摘要与指纹。
2. 基于当前 case 返回历史相似 case。
3. 为每个相似 case 展示：
   - 相似点
   - 差异点
   - 历史结果
4. 至少支持按任务描述与 case 结构双维度检索。

## 9. 非功能需求

- 性能：单事件写入 p95 小于 300ms；单个 case 回放首屏小于 2s
- 可靠性：事件落库成功率大于 99.9%；case 结束后结构化成功率大于 95%
- 可解释性：每个关键决策都必须可追溯到原始事件
- 安全性：默认脱敏敏感参数；命令与文件记录支持遮罩
- 可扩展性：后续可扩展到其他 agent runtime，但 MVP 不要求先实现

## 10. 成功指标

### 10.1 核心验证指标

- 轨迹完整率：一次 case 中关键事件被成功记录的比例
- 决策解释覆盖率：有完整“问题 + 证据 + 约束 + 动作”解释的决策节点比例
- 回放可用率：用户能基于回放理解 case 主要过程的比例
- 先例有效率：用户认为返回的相似 case 对当前任务“有帮助”的比例

### 10.2 体验验证指标

- 单次 case 复盘时间下降比例
- 定位错误决策节点的平均时间
- 用户对“为什么这样做”解释的满意度

## 11. 风险与应对

### 11.1 主要风险

- 采集不完整，导致回放与解释失真
- 决策节点抽取过多，噪音压倒价值
- 解释过于生成式，缺乏证据绑定，失去审计意义
- 相似 case 检索不准，先例价值不成立
- 一开始就做重图谱，工程复杂度失控

### 11.2 应对策略

- 先以事件流为第一事实来源
- 决策抽取优先采用规则 + 少量模型补全
- 解释必须绑定证据引用
- 先做 case 检索，不做复杂多跳图推理
- 底层先用关系型存储与派生索引，不预设图数据库

## 12. 依赖

- OpenClaw 运行时可暴露基础事件或可被外层 hook
- 存在用于摘要与相似度生成的模型能力
- 有最小前端或本地查看界面承载 case 回放

## 13. 开放问题

- case 的结束条件是用户显式结束，还是基于长时间无活动自动关闭
- 模型输入输出要记录到什么粒度，才能兼顾复盘价值与敏感信息控制
- 文件修改应记录 patch、全文快照，还是两者并存
- 先例检索优先以 case 级别还是决策节点级别返回
- 第一版是否需要支持人工补充“这一步为何重要”的标注

## 14. 当前结论

截至 2026-03-09，最合理的 MVP 不是先做企业级 Context Graph 平台，而是围绕本机 OpenClaw 的单 case 闭环，验证以下链路是否成立：

1. 轨迹可采集
2. 决策可抽取
3. 过程可回放
4. 原因可解释
5. 历史可复用

只要这五件事成立，Context Graph 的第一性价值就已经被验证。

## 15. 决策记录

- `2026-03-09`: MVP 锚点收敛为“本机 OpenClaw 单实例真实任务闭环”，不以多场景平台为起点。
- `2026-03-09`: MVP 优先验证 case 级轨迹、决策、解释和先例四项最小能力，不做重平台抽象。
- `2026-03-09`: MVP 底层默认采用事件流 + 结构化决策记录的两层模型，而不是先构建完整图数据库。
