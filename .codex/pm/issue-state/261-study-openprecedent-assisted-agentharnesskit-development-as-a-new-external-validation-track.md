## Summary

Track AgentHarnessKit as a new external validation line for OpenPrecedent so future harness-scaffold development rounds can produce structured research evidence instead of isolated local observations.

## Validated Facts

- issue `#261` exists as a `research` issue
- AgentHarnessKit is a different external repository category than HarnessHub because it focuses on harness infrastructure rather than product delivery
- OpenPrecedent has already been privately activated in the current AgentHarnessKit clone through a local hidden overlay rather than a public repository dependency
- future evidence from that repository should be analyzed separately from the completed HarnessHub studies
- the first AgentHarnessKit wave has already produced four recorded invocations with non-empty `matched_case_ids`
- the observed query reasons so far are `initial_planning` and `after_failure`; `before_file_write` has not appeared yet

## Open Questions

- whether precedent retrieval remains useful when the external project is primarily workflow and guardrail infrastructure
- what kinds of precedent are most often retrieved in a harness-scaffold repository
- whether AgentHarnessKit development will produce enough distinct rounds to justify a closeout study comparable to HarnessHub
- whether later AgentHarnessKit rounds will begin to show stable `before_file_write` usage rather than only planning and recovery usage

## Next Steps

- continue using OpenPrecedent during substantive AgentHarnessKit issue work
- continue recording observations while the sample is small
- archive sanitized evidence once enough rounds accumulate
- keep the issue in progress while the new external validation track is still gathering evidence
